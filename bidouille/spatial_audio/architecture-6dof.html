6DOF Spatial Audio — Architecture

Today the toy tracks 3 degrees of freedom — head rotation from AirPods. This is the design to reach full 6DOF: keep the AirPods rotation, and add the three translation axes so you can physically walk around a sound that stays nailed to a point in the room. Position comes from Ultra-Wideband (UWB) ranging between an iPhone in your pocket and a fixed anchor at the room centre — the same spot as your MacBook, which acts as the calibration origin. And the sound itself can move: strap a tracker to a LEGO car and the source drives around the room in real time (§08).

rotation — AirPods IMU (3DOF) translation — UWB + iPhone (3DOF) origin / calibration — Mac + anchor

01The six degrees of freedom

"6DOF" = the six independent ways a rigid body can move in space: 3 rotations (which way your head points) plus 3 translations (where your body is). Each axis below is tagged with the sensor that supplies it and how trustworthy that source is.

DOFKindMotionSourceQuality (pocket)
Yawrot turn left / rightAirPods IMU · attitudesolid · slow drift
Pitchrot nod up / downAirPods IMU · attitudesolid

Honest limit: UWB nails the horizontal plane (X, Z) but is weak vertically (Y), and a phone in a pocket occludes the antenna so direction (angle-of-arrival) degrades while distance survives. We compensate by fusing UWB with the phone's own inertial dead-reckoning — see §05.

02System architecture — data flow

Three independent producers feed one consumer. The AirPods → Mac path already exists (HeadBridge.app). The new iPhone → UWB anchor path produces position, streamed over the LAN. The web page fuses both into a single moving, turning listener.

03Components

AirPods + HeadBridge rot

exists today · Mac · ws://localhost:8766

The current menu-bar agent. Reads CMHeadphoneMotionManager attitude and streams yaw/pitch/roll + quaternion. Unchanged for 6DOF — it owns the 3 rotation axes.

UWB anchor trans

A self-powered Ultra-Wideband board implementing Apple's Nearby Interaction Accessory Protocol. Sits exactly where the Mac is — the fixed reference the phone ranges against.

RoomBridge (iOS) trans

Runs an NISession to the anchor → distance + direction → 2-D position. Fuses with on-device step/heading dead-reckoning, then streams {x,y,z,conf} to the page over Wi-Fi.

Mac — origin & host org

runs the page + HeadBridge · is the (0,0,0)

Serves index.html, plays the binaural audio through the AirPods, and defines the room origin. "Calibrate at the Mac" zeroes both coordinate frames.

Fusion layer org

in-page · JS

Aligns CoreMotion, UWB and Web-Audio axes into one world frame, then sets the listener's position (UWB) and orientation (AirPods) every frame.

CarBridge trans

new · webcam / UWB tag · ws://localhost:8768

Tracks a moving object (a LEGO car) and streams the source position so the sound itself drives around the room. See §08 — and note an AirTag can't do this.

Web Audio engine rottrans

in-page · PannerNode (HRTF)

The listener moves (UWB) and rotates (AirPods); the source is a fixed point or a moving tracked object (CarBridge). Both ends get full 6DOF.

04The coordinate frames & calibration

The whole trick is making three frames agree: CoreMotion's attitude reference (AirPods), the UWB anchor's local frame, and Web Audio's axes. We force agreement with one ritual at a known spot — the anchor/Mac at the room centre.

Calibration ritual

  1. Stand at the anchor (the Mac, room centre) facing your chosen "front" of the room.
  2. Press Calibrate. Position is zeroed: UWB → (0,0,0). The AirPods attitude at this instant becomes identity forward (re-using the existing yaw/pitch/roll-zero logic).
  3. Now both frames share an origin and a +Z forward axis. UWB (r, θ) maps to world X = r·sinθ, Z = r·cosθ.
  4. Walk. The listener translates with UWB and rotates with AirPods, independently.
  5. Re-zero on drift — step back onto the anchor and Calibrate again (UWB has no long-term drift, but PDR fallback and AirPods yaw do).

05Degradation & fusion (pocket reality)

A pocketed phone wrecks UWB angle but keeps range, and UWB updates slowly (~tens of Hz, with dropouts). So RoomBridge runs a complementary filter:

UWB — absolute, slow, drift-free

Trust range always; trust angle only when confidence is high (phone briefly steady / unobstructed). UWB is the anchor of truth that kills long-term drift.

Dead-reckoning — relative, fast, drifts

CMPedometer steps × stride → distance; compass/deviceMotion → heading. Fills UWB gaps frame-to-frame, then gets snapped back to the next good UWB fix.

Each position message carries a conf field so the page can soften the listener move (or freeze it) when tracking quality drops, instead of letting the sound jump.

07Fusion math — setting the listener

Unlike today's 3DOF hack (fixed listener, source rotated by -yaw), 6DOF uses the textbook model: the source is a fixed world point, and we drive the AudioListener with both streams each frame. 

// position — straight from UWB (metres, origin = anchor)
listener.positionX = pos.x;  listener.positionY = pos.y;  listener.positionZ = pos.z;

// orientation — forward & up vectors derived from AirPods attitude
// (rotate the calibrated forward/up by the head quaternion)
const f = rotateByQuat([0,0,-1], head.q);   // where the face points
const u = rotateByQuat([0,1, 0], head.q);   // head "up" (carries roll)
listener.forwardX = f[0]; listener.forwardY = f[1]; listener.forwardZ = f[2];
listener.upX      = u[0]; listener.upY      = u[1]; listener.upZ      = u[2];

// source — a fixed point, OR streamed live from CarBridge (the LEGO car, §08)
panner.positionX = src.x; panner.positionY = src.y; panner.positionZ = src.z;
Axis caveat (from the existing build): CoreMotion's reference frame and Web Audio's axes don't line up out of the box — a naive quaternion hand-off leaks head-turn into the wrong axis. The calibration in §04 (capturing the attitude basis at the origin and re-expressing forward/up relative to it) is what makes the quaternion path safe here.

08Moving the sound — tracked objects

The source doesn't have to be a fixed point. Strap a tracker to a moving object — a LEGO car — and the sound drives around the room with it, in real time. Architecturally it's just a third producer, CarBridge, streaming the source's (x, z) the same way HeadBridge streams head orientation. Now both ends move independently: you walk and turn, the car carries the sound.

Why not an AirTag? AirTags expose no developer API, update only every few seconds-to-minutes through Find My, and go coarse indoors — useless for live motion. Track the car a way you can actually read at frame rate:
